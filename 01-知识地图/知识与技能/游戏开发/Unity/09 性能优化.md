# 性能优化

性能优化主要聚焦在**内存**、**CPU**、**GPU**三大方向上。粗略划分如下：

- **内存**：资源内存占用、引擎模块自身内存占用、托管堆内存占用等。

- **CPU**：引擎模块性能开销、自身代码开销。

- **GPU**：OverDraw、显存带宽等。

## 优化方案

有各种各样的处理手段，简要列举如下：

### （1）内存处理手段

- 在 Unity 中，前置堆内存分配，尽量降低碎片化；

- 查验内存泄露；

- 资源预处理；

- 字符串拼接、配置表优化、对象池等。

### （2）CPU处理手段

- 在 Unity 中，采用静态批处理、动态批处理、遮挡剔除、远近剔除、AOI、合理划分图集、动态图集、美术模型顶点控制、特效面数控制、GPU Instancing、UGUI 使用的注意事项等，来控制 DrawCall；

- 优化业务代码执行效率。

### （3）GPU处理手段

- **OverDraw**：单个像素在单位时间内被多次渲染。围绕 OverDraw 的基本概念进行优化；

- 其他 GPU 相关优化，如显存带宽等。

## 第一篇：内存篇

内存的开销主要有以下三部分：

- 资源内存占用

- 引擎模块自身内存占用

- 托管堆内存占用

### 资源内存占用

在较为复杂的大中型项目中，资源的内存占用通常占到总体内存的 70%以上。资源主要分为纹理（Texture）、网格（Mesh）、动画片段（AnimationClip）、音频片段（AudioClip）、材质（Material）、着色器（Shader）、字体资源（Font）以及文本资源（Text Asset）等。其中，纹理、网格、动画片段和音频片段往往是较大内存开销的来源。

#### 一、纹理

纹理资源是大部分游戏项目中占用最大内存的资源。以一个 6 万面片的场景为例，网格资源最大不过 10MB，但一个 2048x2048 的纹理可能直接占用 16MB。因此，纹理的合理使用至关重要。注意以下几个方面：

- **纹理格式**

- 根据硬件种类选择支持的纹理格式，比如 Android 平台的 ETC、iOS 平台的 PVRTC、Windows PC 上的 DXT 等。

- 色阶问题：有损压缩格式（如 ETC、PVRTC）在色差较大时会造成色阶问题，使用 RGBA32/ARGB32 格式可改善效果，但会显著增加内存占用。

- ETC1 不支持透明通道：对于 OpenGL ES 2.0 的设备，ETC1 不支持 Alpha 透明通道，建议将透明贴图分拆为 RGB24 和 Alpha8 两张贴图分别储存并通过 Shader 渲染。

- 新设备可使用 ETC2、ASTC 等格式，支持透明通道且压缩比更好。

- **纹理尺寸**

- 纹理尺寸越大，内存占用越大。应尽可能使用较小的纹理尺寸，512x512 的纹理已足够则不应使用 1024x1024 的纹理。

- **Mipmap 功能**

- Mipmap 旨在降低渲染带宽压力，提升渲染效率，但会增加 1.33 倍的内存占用。对于3D场景建议开启 Mipmap，但 UI 纹理一般不需要开启。

- **Read & Write**

- Unity 中“Read & Write”功能默认关闭，但部分项目仍会错误地开启该选项，导致纹理内存增大一倍。

#### 二、网格

网格资源在复杂游戏中内存占用较高。注意以下几点：

- **Mesh 数据优化**

- 深度优化项目中经常发现 Mesh 数据中包含大量不必要的 Color、Normal、Tangent 数据，这会增加文件体积和内存占用。

- **Draw Call Batching 对内存的影响**

- Mesh 拼合时，若某些 Mesh 含有多余数据，会为所有拼合的 Mesh 添加该属性，造成内存浪费。建议检查 Shader 渲染时是否需要这些数据。

#### 三、引擎模块自身占用

引擎自身存在大量内存开销，如 GameObject、Component（如 Transform）、ParticleSystem、MonoScript 以及各种 Manager（如 SceneManager、CanvasManager）。其中，WebStream 和 SerializedFile 占用较大。

- **WebStream**：主要由 AssetBundle 加载资源引起，需注意是否存在未清理的 AssetBundle。

- **SerializedFile**：使用 LoadFromCacheOrDownload、CreateFromFile 时产生，需注意管理。

#### 四、托管堆内存占用

托管堆内存由 Mono 分配和管理，内存一旦分配不会归还系统。因此需避免以下问题：

- **高频率的内存分配**：避免在高频函数中分配内存，例如 Update 或 FixedUpdate。

- **Log 输出**：应严格控制 Log 输出，避免不必要的堆内存分配。

- **UIPanel.LateUpdate**：NGUI 中 CPU 和堆内存开销较大的函数，需针对 UI 网格重建进行优化。

## 第二篇：CPU篇

Unity 移动游戏中，CPU 性能开销主要分为两类：引擎模块性能开销和自身代码性能开销。

### 一、渲染模块

渲染模块是游戏中 CPU 性能的主要消耗，优化重点在以下两点：

- **降低 DrawCall**

- DrawCall 越高，CPU 开销越大。通过减少材质种类、使用 Draw Call Batching 来减少其数量。注意 Draw Call 并非越低越好，还需平衡总线带宽。

- **简化资源**

- 过量网格资源、纹理资源等都会增加 CPU 负担，应尽量简化。

### 二、UI 模块

UI 模块中，**UIPanel.LateUpdate** 是 CPU 开销最大的函数。优化策略包括：

- 将动态 UI 元素与静态 UI 元素分离到不同的 UIPanel 中；

- 控制同一 UIPanel 中动态 UI 元素的数量，尽量减少 UI 网格重建次数。

### 三、加载模块

场景切换时 CPU 开销较高，主要体现在场景卸载和场景加载：

- **场景卸载**

- 引擎自动调用 Destroy 和 Resources.UnloadUnusedAssets，后者 CPU 开销较大。

- **场景加载**

- 资源加载和 Instantiate 操作会消耗大量 CPU，建议优化资源依赖关系并使用预加载。

### 四、代码效率

逻辑代码开销集中在少数函数上，遵循“二八原则”，即80%的开销集中在20%的函数上。使用 Profiler 找出性能瓶颈函数，进行针对性优化。

## 第三篇：GPU篇

GPU 瓶颈主要来自以下几个方面：

- **填充率**：每秒渲染的像素数量。

- **像素复杂度**：如动态阴影、光照、复杂的 Shader 等。

- **几何体复杂度**：顶点数量。

- **显存带宽**：影响 GPU 的资源存取效率。

### 优化策略

- **减少顶点数量、简化计算复杂度**

- 保持材质数量少，使用纹理图集、光照纹理、LOD、遮挡剔除（Occlusion Culling），选择简单的 mobile 版 Shader。

- **优化显存带宽**

- 使用 ETC1、PVRTC 等压缩格式，使用 Mipmap 来优化显存带宽，尽管会增加内存占用，但可以根据实际情况选择合适的小图进行渲染。
