# 系统模块

# 1.  地图系统

小地图是属于主UI界面的，只要加载过一次，就是全局的UI，它就会一直存在，不会随着地图的切换而销毁。并且主UI界面的层级都比较低，所以哪怕是回到选角色或者登入场景，主UI界面都会被遮盖掉，不会影响正常交互。

登入并且选完角色进入游戏后，每个地图都会有个负责通知场景变化的脚本（MapController）存在，它会告诉地图管理器（MapManager）当前地图的一些信息，比如地图的长和宽等。然后，地图管理器就会去调用小地图的UI控制脚本（UIMiniMap），让他刷新当前场景对应的资源，比如场景大小和地图图片等。

同时，小地图的UI控制器（UIMiniMap）还会实时显示当前玩家所在的地图位置，玩家永远是在地图的中心位置。原理是通过计算玩家所在的世界坐标的 xz 与包围盒的左下角起点的世界坐标 xz 的差值，再与包围盒的宽高形成一个比例，得到的宽和高的比例值就是小地图 xy 的锚点值，将这个锚点值设置为小地图的锚点，并且设置小地图为居中模式，这样玩家就会一直在地图的中心，同时地图也在不断更新实时位置。

# 2. NPC 系统

每个 npc 都有自己的 NPCController，当玩家点击 NPC 时，会先判断是否在交互范围内，如果超出则会先移动向 NPC，当进入交互距离的时候，就会由 NPCManager 判断当前交互的内容是什么，任务交互的优先级是高于功能交互的，因此 NPCManager 会先调用 QuestManager 判断当前 NPC是否存在正在进行中的任务，如果存在则先触发任务交互，反之，则触发 NPC 对应的功能交互，例如打开当前 NPC 对应的商店界面。

# 3. 道具系统

道具系统负责管理玩家的所有道具信息，其他系统也都是通过道具系统获取到道具的具体数值，当道具数量发生变化时，也是由道具系统进行通信交互。不过金币这个特例由 StatusService 管理，它和玩家的状态变化放在一起处理。

# 4. 背包系统

当玩家选择完角色进入游戏后，会先初始化背包管理器（BagManager），这个背包的数据是由服务器端发送而来的，数据都是存储在 SQL Server 中的。传过来的数据会保持之前的道具位置摆放，玩家可以通过点击整理按钮，将相同的道具堆叠在一起，同时按照一定的顺序进行摆放整理，原理是先清楚所有显示的道具，再遍历生成道具系统中的所有道具，当某个道具达到单格最大道具限制时，会将剩余的道具数量存储到下个格子当中。当玩家购买或使用某个道具时，会先由道具系统的 ItemService 与服务器进行交互，当信息传回时，除了更改道具数据，还会响应给所有关注该信息的 UI 模块，比如购买时，会响应给商店，刷新剩余的可购买数据，使用时，则刷新背包的 UI 界面。

# 5. 商店系统

当玩家选择完角色进入游戏后，会先初始化商店管理器（ShopManager），将打开商店相关的方法注册到 NPC 的交互委托中，这样后续交互只需调用打开商店的委托即可。通过委托打开商店之后，会有一个专门管理 UI 视图的 UIShop，UIShop 会根据当前的 NPC 打开对应的商店，并且会生成对应的道具 UIShopItem，当我们点击某个道具时，UIShopItem 会反馈给 UIShop 自己被选中了，当购买某个道具时，UIShop 会反馈给 ShopManager，并由 ShopManager 告知 ItemService ，让 ItemService 与服务器去进行通信，当结果返回回来时，直接调用订阅了委托的 UIShop进行 UI 界面刷新，比如金钱减少等。

实际上，金钱的变化和道具的变化不太一样，金钱的变化由 StatusService 管理，而道具的变化由 ItemService 进行管理。

# 6. 装备系统

当玩家选择完角色进入游戏后，UserService 会先将玩家之前存储在数据库中的装备数据传给装备管理器进行初始化。

当玩家点击装备的 UI 按钮时，会打开装备面板，同时管理该 UI 的 UICharEquip 脚本会生成对应的 UIEquipItem，包括符合穿戴条件但还未穿戴的装备和已经穿戴的装备。

当我们选中某个未穿戴的装备时，它会将该装备的信息传给UICharEquip，当我们点击装备的时候，UICharEquip 会调用 EquipManager，再由EquipManager 调用ItemService 去与服务器进行通信。返回的结果会由ItemService 调用EquipManager，再由EquipManager 通过调用事件的方式触发UICharEquip 刷新 UI 的方法。卸下装备同理。

# 7. 任务系统

当玩家选择完角色进入游戏后，UserService 会先将玩家之前存储在数据库中的任务数据传给任务管理器（QuestManager）进行初始化。

当玩家点击任务的 UI 按钮时，会打开任务面板，同时管理该 UI 的UIQuestSystem 会生成所有的已接任务，每个生成的物体对应一个 UIQuestItem，我们还可以点击可接任务，那么UIQuestSystem 就会生成所有的可接取任务，这个原理是读取配置表中的信息，将达到可接取条件的任务与已接任务进行对比，如果达到接取条件，并且还没有接取，则将显示在面板上。

当我们点击某个任务的时候，UIQuestItem 会通过委托的方式反馈给UIQuestSystem，让他刷新 UI，显示该任务的详细信息，该任务的详细信息由UIQuestInfo 控制显示。并且，还可以通过点击追踪，调用玩家的 PlayerInputController， 让玩家自动导航到该任务对应的位置。

与 NPC 进行交互的时候，会先检查该 NPC 是否存在正在进行的任务，如果存在，则先触发剧情任务对话，由 QuestManager 打开对话面板UIQuestDialog 并且初始化面板信息。

# 8. 好友系统

当玩家刚进入游戏时，好友系统的通信层（FriendService）就开始初始化了，当玩家选择完角色进入游戏后，UserService 会先将玩家之前存储在数据库中的好友数据传给好友管理器（FriendManager）进行初始化。

当玩家点击好友 UI 按钮时，会打开好友面板，同时管理该 UI 的UIFriends 会根据好友管理器（FriendManager）中的数据生成对应的好友列表，每个好友项 UI 由UIFriendItem 管理 。当调用好友管理器（FriendManager）中的 数据属性 get 时，会对数据进行一次排序，同一等级下，比较是否在线，同一状态下，比较等级大小，在线状态优先级高于等级，排序完成后，管理好友 UI 界面的UIFriends 直接顺序生成好友信息即可。

还可以通过查找，筛选好友，原理是利用 Linq 语句的 Where 进行筛选，返回符合筛选条件的列表，并显示出来。

添加好友时，可以输入 id 或用户名进行添加，首先是由 A 客户端发送 添加好友的请求FriendAddRequest-> 服务器端接收到后进行数据校验，如果不存在或者不在线，则发回失败响应FriendAddResponse。反之，服务器进行转发，发给对应的 B 客户端 添加好友的请求FriendAddRequest   -> B 客户端接收到 添加好友的请求 后，会弹出添加好友界面，做出选择后 向服务器发出 好友处理响应 FriendAddResponse -> 服务器端接收到 好友处理请求后 向 A 和 B 两个客户端都发送好友处理响应FriendAddResponse，如果是申请者，则弹出是否添加成功，如果是接受者，则弹出添加成功或拒绝成功。

删除好友时，选择好友即可，无论该玩家是否在线都可成功互相删除，若对方在线，则将删除结果同步给对方客户端，如果不在线，则直接在数据库找到对方的好友列表，并将自己删除。

# 9. 组队系统

当玩家刚进入游戏时，组队系统的通信层（TeamService）就开始初始化了。组队系统中的组队申请和好友系统中的好友申请的流程是一致的，先由 A 客户端发送组队请求TeamInviteRequest，服务器端接收到请求后，先检验数据是否正常，然后再转发给 B 客户端该请求TeamInviteRequest，B 客户端对此申请做出回应后，会返回 TeamInviteResponse 给服务器端，服务器端再转发给 A 和 B 客户端TeamInviteResponse，如果是申请者，则弹出是否申请成功，如果是接受者，则弹出添加成功或拒绝成功。最初的申请者将成为队长，非队长不具备组队请求的权限。

还可以转让队长给其他队员，并刷新所有客户端的组队界面，这里的刷新不是同步的，而是并发数据，等待其他的信息一起并发给其他客户端。

还可以直接离队，如果是队长，则会将队长之位转给第二个进入队伍的成员，并刷新所有客户端的组队界面，这里的刷新也不是同步的。
